энергоаудита. Каждый файл содержит и **входы** (паспорт насоса/ЭД, режим), и
**эталонные выходы** (УРЭ, КПД, напор, годовые потери). Модель прогоняется на
извлечённых входах; выход сверяется с эталоном.

## Разбор расхождений

### 1. КНС-ОПУ НА-1/2/3 — ошибка в эталонном файле (8 из 8 ✗ — здесь)
В файле `КНС -ОПУ расчет.xlsx` номинальный КПД **«залип» на 0,5766** для трёх
разных насосов:

| Агрегат | Насос | η_нас.ном паспорт | КПД_ном в файле |
|---|---|---|---|
| НА-1 | ЦНС 63-1400 | 0,60 → η_ном 0,577 | 0,559 |
| НА-2 | ЦНС 80-1400 | 0,63 → η_ном 0,605 | **0,5766** |
| НА-3 | ЦНС 180-1422 | 0,76 → η_ном 0,736 | **0,5766** |

Модель берёт корректный паспорт каждого насоса → расходится с эталоном на
ΔW_кпд +22…+76 %. **Причина (г): неполнота/ошибка исходных данных. Модель корректнее.**
(Файл — черновик выезда 08.04.2026; вероятно, не дозаполнен.)

### 2. КНС-14 НА-1 (ГНУ REDA, ЭЦН) — методическое отличие
УРЭ расчётный −3,0 %, КПД ном модель 0,617 vs эталон 0,598. Инженер применил для
ЭЦН-установки **дерейтинг η_ном** (учёт КПД ПЭД и кабеля ~0,92), которого нет в
простой формуле (14). В допуске ±5 %. **Причина (в): различие методики для ЭЦН.**
Уточнение: для погружных насосов добавить компоненту КПД ПЭД/кабеля.

### 3. ΔW_дрос — системное расхождение «методика ↔ практика инженеров» (уточнено 02.07.2026)
Формула (45) методики делит на **η_ном**; все инженерные xlsx делят на **η_факт**.
Проверено на 8 агрегатах 8 объектов: модель·η_ном/η_факт = эталон с точностью 0,1 %
(тест `test_dw_throttle_eta_fact_convention`). Модель следует букве методики, поэтому
её ΔW_дрос систематически ниже эталона на η_факт/η_ном (−3…−35 %, растёт с износом
насоса); при Δp_задв ≈ 0,01–0,03 МПа (КНС-10 НА-1, КНС-154 НА-2) величина к тому же
абсолютно несущественна (единицы–тысячи кВт·ч против сотен тысяч у ΔW_кпд).
Разбор и вопрос заказчику — `docs/audit_findings.md` §В2.

### 4. КНС-155т НА-2 — определение Q для гидромощности (КПД факт)
Эталон (xlsx) и текстовый отчёт **согласованы между собой**: КПД факт = 0,661, K_з = 0,296.
Модель даёт КПД факт = 0,580, K_з = 0,953. Разбор:
- **УРЭ факт совпадает точно** (5,026 кВт·ч/м³) — он считается из W и Q_сут напрямую.
- КПД факт = P_гидр/P_эл. P_эл = W/T = 987 кВт (модель и эталон совпадают; отсюда K_з модели
  0,953 = 987/(1000/0,965) — это и есть корректный K_з; эталонные 0,296 ему противоречат).
- P_гидр эталона = 652,7 кВт при Δp = 10,49 МПа подразумевает **Q ≈ 224 м³/ч**, тогда как
  суточный Q = Q_сут/T = 4715/24 = **196,5 м³/ч** (его же даёт h_факт, совпадающий с моделью).
- Итог: эталон посчитал КПД/K_з через **мгновенный** Q≈224, модель — через **суточный** 196,5.
  Это различие определения Q, а не ошибка модели; внесено в `KNOWN_DEVIATIONS` и
  `KNOWN_MEASURED_KPI_DEVIATIONS` (`tests/verification/test_objects.py`).

# Трёхсторонняя сверка с текстовыми отчётами (Задача 1, расширение)

Помимо числовых «… расчет.xlsx» каждый объект описан **текстовым отчётом**
энергоаудита («Промежуточный отчет НТУ по …», `.doc`/`.docx`) — независимым
«человеческим» эталоном с прозой, таблицами и ТЭО. Парсер `ingest/report_doc.py`
извлекает **и таблицы, и числовые утверждения из прозы**, в т.ч. относительные:
«КПД снижен на X п.п.», «УРЭ больше расчётного на X», «рабочая точка ниже Q-H на N м (Z %)».

Воспроизвести: `python -m ppd_audit.verify` → трёхсторонняя таблица +
`data/generated/reconciliation_reports.{csv,md}`. Бинарные `.doc` конвертируются
LibreOffice (`soffice`) в `data/reports/<id>.docx`.

## Конвертация .doc → .docx

Старые отчёты (КНС-155/10/154, КНС-ОПУ, КНС-14) — в бинарном Word `.doc`,
который python-docx не читает. Конвертер `ingest/convert.py` вызывает
`soffice --headless --convert-to docx` и кэширует результат в `data/reports/`
(оригиналы в «НТУ цифровая платформа» не трогаются). Берётся **самая поздняя**
итерация отчёта по каждому объекту. Резервные конвертеры (antiword/textract) не
используются — они теряют структуру таблиц.

## Два шаблона отчётов

- **2024** (КНС-25/155/10/154): агрегаты — столбцы значений, один замер, проза в абзацах/ТЭО.
- **2026** (КНС-ОПУ, КНС-14): на каждый агрегат — таблица с **несколькими датами**;
  значение отчёта берётся как **диапазон [min, max] по датам** (модель/xlsx считаются
  для одного снимка → проверяем попадание модели в диапазон). Проза даёт диапазоны («9,1…9,6 п.п.»).

## Итог трёхсторонней сверки

**116 строк, 70 пар «xlsx↔отчёт». Согласованность источников ≈ 86 %.**

> Для согласованных объектов (КНС-25, КНС-155, и большинство КНС-10/154) модель
> совпадает **и с инженерным xlsx, и с текстовым отчётом**, включая относительные
> утверждения прозы: напр. КНС-25 «КПД снижен на 2,59 п.п.» (модель 2,59),
> «УРЭ выше расчётного на 0,14» (модель 0,14); КНС-155 «4,384 п.п. / 0,379 / 54 м (4,78 %)».

## Разбор расхождений источников (xlsx ≠ отчёт)

### КНС-ОПУ — разные снимки и привязка агрегатов
Отчёт 05.05.2026 — ряд из 3 дат (март–апр 2026), xlsx — отдельный снимок; измерения
ЦНС-63/ЦНС-80 в источниках привязаны к разным агрегатам, η_ном в xlsx «залип» на 0,577
(см. выше). Модель верно воспроизводит **xlsx** (модель↔xlsx ✓ по измеряемым KPI),
расхождение — между двумя эталонами инженера. **Флаг «источники расходятся».**

### КНС-14 АН — инвертирован порядок/тип агрегатов
В xlsx НА-1 = ГНУ REDA (ЭЦН), НА-2 = СИН (плунжер); в отчёте REDA подписан как № 2.
По измеряемому УРЭ значения сходятся (одна и та же физика), но ΔW_кпд по НА-2 расходится
из-за разной привязки. **Флаг источников.**

### КНС-10 БН НА-1 — внутренняя несогласованность отчёта
В отчёте р_вых в «исходных данных» (8,92 МПа → напор 703 м) ≠ «технологических
параметрах» (949 м). xlsx/модель считают по 8,92 (703 м), отчёт в таблице тех.параметров
показывает 949 м. **Флаг: данные внутри отчёта противоречивы.**

### КНС-154 БН НА-2 — округление η_ном
Отчёт η_ном 0,756, xlsx/модель 0,743 → «КПД снижен» 10,3 п.п. (отчёт) vs 9,0 (модель).
В пределах ⚠; причина — округление номинального КПД в отчёте.

Все расхождения «модель↔отчёт» (✗) сведены в реестр `KNOWN_SOURCE_DISAGREEMENTS` /
`KNOWN_MODEL_REPORT_FAILS` (`tests/reports/test_reports.py`) — новые, необъяснённые,
роняют сборку. Это и есть цель: **трёхсторонняя сверка ловит расхождения между
источниками и внутри отчётов**, а не маскирует их.
